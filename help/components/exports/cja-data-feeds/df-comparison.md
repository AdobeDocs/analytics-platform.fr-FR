# Comparaison des flux de données dans Customer Journey Analytics et Adobe Analytics

Les flux de données dans Customer Journey Analytics et Adobe Analytics vous permettent d’exporter des données brutes vers des plateformes tierces. Si vous avez déjà utilisé des flux de données dans Adobe Analytics, utilisez les informations suivantes pour comprendre les différences de fonctionnalités disponibles, de comportement de traitement des données, etc.

## Fonctionnalité Comparer les flux de données

| **Options et concepts de configuration** | **Description** | **Customer Journey Analytics** | **Adobe Analytics** |
|---------|----------|---------|---------|
| Entrée de données | Type de données qui peut être collecté et inclus dans les flux de données. | Prend en charge l’entrée de données cross-canal, y compris les données web, les données de centre d’appels, les données de point de vente, etc. | Prend uniquement en charge l’entrée de données web et mobile. |
| Traitement des données | Les données doivent souvent être traitées avant d’être utiles pour la création de rapports, et ce traitement affecte la fonctionnalité des flux de données. Les données sont traitées à différentes étapes, selon que vous utilisez Customer Journey Analytics ou Adobe Analytics. | Les données sont traitées au moment du rapport. De ce fait, de nombreuses fonctions de création de rapports de Customer Journey Analytics peuvent être utilisées pour modifier les données historiques, comme le groupement, les champs dérivés, la préparation des données, la segmentation et la fenêtre de création de rapports. | Les données sont traitées au moment de la collecte. Par conséquent, les fonctionnalités de création de rapports telles que les règles de traitement, les règles VISTA, l’assemblage et la segmentation n’affectent pas les données historiques. |
| Fenêtre de flux de données/fréquence de diffusion | Détermine :<ul><li>Fréquence d’envoi du flux de données</li><li>Fenêtre temporelle incluse dans le flux de données<p>En d’autres termes, une fois que la fenêtre de flux de données se termine et que le flux de données est envoyé, une nouvelle fenêtre de flux de données commence.</p></li></ul><p>Les options disponibles sont les suivantes :</p><ul><li>**Quotidien** : les flux contiennent l’équivalent d’une journée complète de données, de minuit à minuit dans le fuseau horaire de la suite de rapports. Utilisez cette option pour les données de renvoi ou historiques, ou pour les flux continus.</li><li>**Par heure** : les flux contiennent l’équivalent d’une heure de données. Utilisez cette option pour poursuivre les flux.</li></ul> | Flux horaires et quotidiens.<p>La fenêtre des flux de données est incluse dans la **[!UICONTROL fenêtre Rapports]**</p>  <p>Exemple : 1 heure</p> | Flux horaires et quotidiens. En outre, les flux de 15 minutes sont possibles, mais ne sont pas disponibles par défaut. </p><p>Les accès tardifs peuvent être configurés en fonction de l’**[!UICONTROL intervalle de recherche en amont]**.</p><p>Exemple : 1 heure</p> |
| Inclusion des accès en retard | Les accès tardifs sont des accès qui arrivent pendant la fenêtre actuelle du flux de données, mais dont la date et l’heure correspondent à une fenêtre précédente du flux de données (par exemple, via des accès horodatés ou des sources de données). Lorsque les accès arrivent en retard, ils sont regroupés dans le fichier de flux de données suivant qui est envoyé, même si leurs horodatages appartiennent à une fenêtre antérieure. | Les accès tardifs qui se produisent dans la **[!UICONTROL fenêtre de création de rapports]** sont toujours inclus. <p>Les accès sont automatiquement réorganisés par horodatage dans la sortie (par exemple, les accès tardifs peuvent apparaître avant les accès ponctuels si leurs horodatages sont antérieurs). Comme il n’existe aucun flux de modification, les valeurs d’origine sont conservées. | Les accès en retard peuvent être inclus ou exclus, et peuvent être configurés avec l’option de configuration **[!UICONTROL Accès en retard]**.<p>L’intervalle de recherche en amont pour ces accès est configuré à l’aide de l’option de configuration **[!UICONTROL Intervalle de recherche en amont]** disponible à cet effet spécifique.</p><p>Les accès s’affichent dans l’ordre dans lequel ils sont reçus ; ils ne sont pas réorganisés selon l’horodatage.</p> |
| Accès dans le désordre | H | Le problème de désordre est évalué par personne ; CJA peut accepter les modèles d’arrivée en flux continu + par lots et les réorganiser une fois que les données arrivent dans la bonne séquence temporelle. CJA n’attend pas de confirmer qu’il dispose de toutes les données pour la fenêtre de création de rapports avant l’envoi ; il envoie ce qui est arrivé au cours de l’heure/la journée (ce qui peut inclure les accès tardifs dont la date et l’heure sont antérieures, dans la fenêtre de création de rapports). <p>**Important** : le consommateur final de vos données de flux de données doit être en mesure de gérer les horodatages dans le désordre, car les données de flux de données peuvent inclure des accès tardifs avec des horodatages antérieurs à la fenêtre du flux de données.</p> | |
| Intervalle de recherche en amont (Adobe Analytics uniquement) | Limite pour les accès en retard ou dans le désordre diffusés en dehors de la **[!UICONTROL fréquence de diffusion]** sélectionnée (par exemple **[!UICONTROL Heure]** ou **[!UICONTROL Jour]**). Pour être incluses, les visites doivent commencer après cette limite. Les visites qui commencent avant la limite et se terminent dans l’intervalle de recherche en amont ne sont pas incluses.<p>Non utilisé pour la persistance, les sessions ou les dimensions (elles sont incluses dans les données brutes collectées).</p> | Inclus dans la fenêtre **[!UICONTROL Reporting]**. | Pris en charge<p>Exemple : 23 heures</p> |
| Créneau de rapport (Customer Journey Analytics uniquement) | La fenêtre temporelle qui comprend :<ul><li>Fenêtre actuelle du flux de données (heure ou jour le plus récent, tel que sélectionné dans le champ **[!UICONTROL Fréquence de diffusion]**).</li><li>Durée spécifiée avant la fenêtre actuelle du flux de données qui autorise les accès tardifs ou dans le désordre.</li></ul> <p>Requis pour les sessions, la persistance et les segments.</p><p>Non utilisé pour les dimensions. Les dimensions sont contrôlées par dimension en fonction de l’attribution et de l’expiration de la dimension. Les recherches en amont dans Dimension ne peuvent pas dépasser la fenêtre de création de rapports.</p> | Pris en charge<p>Exemple : 24 heures</p> | S.O.<p>Voir « Intervalle de recherche en amont ».</p> |
| Fenêtre de renvoi initiale |  | Exemple : 7 jours | Exemple : 7 jours |
| Accès |  | Seul l’accès 5 se trouve dans la fenêtre du flux de données. Cependant, comme la fenêtre de création de rapports inclut également les accès 4 et 3 (qui sont des accès tardifs avec horodatages d’une fenêtre de flux de données précédente), ils sont également inclus dans la fenêtre de flux de données actuelle.<p>Les accès sont réorganisés dans le flux de données en fonction de leur horodatage, comme suit : accès 3, accès 4 et accès 5.</p> | Seul l’accès 5 se trouve dans la fenêtre du flux de données. Cependant, comme une recherche en amont est configurée et qu’elle inclut l’accès 4 et l’accès 3 (qui sont des accès tardifs avec horodatages provenant d’une fenêtre de flux de données précédente), ils sont également inclus dans la fenêtre de flux de données actuelle. (Si une recherche en amont n’a pas été configurée, seul l’accès 5 est inclus dans le flux de données.) <p>Les accès sont affichés dans le flux de données dans l’ordre dans lequel ils ont été reçus, comme suit : accès 4, accès 3 et accès 5.</p> |
| Accès en retard/accès dans le désordre | (Dans AA : les données dans le désordre signifient : dans le désordre par visiteur. Dans CJA : cela signifie « dans le désordre » par personne. Si vous nous envoyez des données qui ne sont pas en ordre par personne, ce serait le cas si vous définissez l’horodatage. Vous pouvez définir l’horodatage de deux manières : Adobe peut définir l’horodatage en fonction de la date et de l’heure de réception des données. Ou vous pouvez le régler vous-même. Si vous définissez les dates et heures et que vous nous envoyez des données qui ne sont pas dans le bon ordre, cela gâche les choses dans AA. Dans AA, les données doivent être classées par visiteur. Nous avons besoin du bon ordre des événements. Mais dans CJA, les horodatages des données n’ont pas d’importance. CJA n’attribue pas d’horodatage à un accès. Cela se fait en amont. CJA réorganise les données une fois qu’elles arrivent, de sorte que tout soit dans la bonne séquence temporelle. Ensuite, nous pouvons procéder au traitement de la période de rapport. Cela signifie que vous pouvez avoir à la fois des données en flux continu et des données par lots. Ça n&#39;a pas d&#39;importance. Au moment où il arrive, nous le réorganisons et il devient ainsi en ordre par personne. Ainsi, dans CJA, nous vous donnerons toutes les données que nous avons reçues au cours de la dernière journée ou heure, mais elles se limitent au début de la période de rapport. Il est probable qu’une grande partie des données que vous obtenez au cours d’une journée ou d’une heure appartient à ce jour ou à cette heure. Si vous ne faisiez que traiter des données par lots à partir d&#39;un centre d&#39;appels, c&#39;est ce que vous obtiendriez. Dans CJA, les données peuvent entrer et le moment où elles sont entrées n’a pas d’importance. Le client du flux de données doit donc être en mesure de gérer cela de son côté. Ainsi, peu importe où ils placent les données, ils doivent tenir compte du fait que les dates et heures peuvent être partout. Cela peut représenter un défi pour certains clients. Ils doivent le savoir. Doit être en mesure de gérer les données dans le désordre par personne. Ça n&#39;a pas d&#39;importance pour les gens. ) Accès qui auraient dû être inclus dans un flux de données précédent, mais qui sont arrivés tardivement pour une raison quelconque (par exemple, via des accès horodatés ou des sources de données). <p>Ces accès tardifs sont inclus dans le flux de données actuel au moment de leur arrivée, même si leurs horodatages se trouvent dans une fenêtre de flux de données précédente. Chaque fois qu’un flux de données traite des données, il examine les accès tardifs arrivés et les transfère par lots dans le fichier de flux de données suivant envoyé.</p> | Les accès tardifs qui se produisent dans la **[!UICONTROL fenêtre de création de rapports]** sont toujours inclus. <p>L’intervalle de recherche en amont de ces accès tardifs est contrôlé par l’option de configuration **[!UICONTROL Intervalle de création de rapports]**.</p><p>Les accès sont automatiquement réorganisés en fonction des horodatages ; les valeurs d’origine sont conservées (aucun flux de modification).</p> | Peut être inclus ou exclu. Configurable avec l’option de configuration **[!UICONTROL Accès tardifs]**.<p>L’intervalle de recherche en amont pour ces accès est configuré à l’aide de l’option de configuration **[!UICONTROL Intervalle de recherche en amont]** disponible à cet effet spécifique.</p><p>Les accès s’affichent dans l’ordre dans lequel ils sont reçus ; ils ne sont pas réorganisés selon l’horodatage.</p> |
| Segmentation |   | Les segments que vous configurez dans Customer Journey Analytics peuvent être appliqués aux flux de données ou à la vue de données.<p>(Personne, Session, Événement) Les conteneurs Personne utilisent l’ensemble de la fenêtre de création de rapports ; ils peuvent développer la sortie de l’accès PDF en fonction de la logique du segment.</p> | Les segments ne peuvent pas être appliqués. |
| Groupement | (pourrait être inclus dans la section traitement des données à la place) | Le groupement peut être utilisé pour joindre plusieurs jeux de données (pour prendre en charge l’analyse cross-canal). | Le groupement n’est pas pris en charge. |
| Schéma | (pourrait être inclus dans la section traitement des données à la place) | Utilise des champs hiérarchiques structurés, contenant potentiellement plusieurs dimensions et mesures dans une seule colonne (tableaux ; tableaux dans les tableaux). | Utilise une liste de produits avec une longue chaîne de champs propriétaires séparés par des points-virgules. |
| Recherches | (pourrait être inclus dans la section de traitement des données à la place) Les recherches dynamiques vous permettent de recevoir des fichiers de recherche supplémentaires dans votre flux de données qui, autrement, ne seraient pas disponibles. | Non nécessaire, car les dimensions du navigateur sont incluses dans la vue de données. Terme plus large qui inclut les mêmes éléments qu’AA, mais inclut également les classifications et les jeux de données de recherche. Vous pouvez créer n’importe quel jeu de données de recherche et l’appliquer à CJA. Et vous n’obtenez pas de fichiers distincts ; ils sont exposés sous forme de dimensions. Vous n’ajoutez pas de fichier de recherches, vous choisissez simplement les dimensions à utiliser. | Utilisé pour faire correspondre un nombre d’une colonne de flux de données à une valeur réelle. Spécifiques à un certain ensemble d’éléments (navigateur, système d’exploitation, appareil mobile) et appliqués sous la forme d’un fichier distinct fourni avec le flux de données. |
| Définition de session | (pourrait être inclus dans la section de traitement des données à la place) Il ne s’agit pas d’une option de configuration dans le flux de données. Mais vous la définissez dans la vue de données. | Défini dans la vue de données | Défini au moment de la collecte. |
| Mesures calculées |   | Non disponible | Non disponible |
| Couverture de la fenêtre de création de rapports dans la section Sortie PDF - Fusionner avec les accès arrivés en retard | Dans CJA, nous ne faisons pas le suivi pour savoir si nous avons reçu toutes les données pour la période de rapport avant d’envoyer le flux de données. Dans AA, nous faisons le suivi de toutes les données qui entrent, puis nous les envoyons quand nous savons que nous les avons toutes. Comme nous en assurons le suivi, nous pouvons donner au client la possibilité d’inclure ou d’exclure les accès tardifs. | Le PDF ne génère qu’une partie de la fenêtre de création de rapports (uniquement la fenêtre actuelle), même si la logique de session/segment peut examiner la fenêtre plus large. | Toute la sortie DF est limitée à la fenêtre fixe (heure/jour/15-min). |
